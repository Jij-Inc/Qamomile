# Qamomile v0.12.5

Qamomile v0.12.5では、`PCEConverter`と`PCEEncoder`によるPauli Correlation Encoding(PCE)を追加しました。これにより、二次の最適化ワークフローで多くのスピン変数を少ない量子ビット上の`k`-body Pauli correlatorとしてエンコードできます。QURI Partsのサンプリングには再現性のためのseedを渡せるようになりました。さらに、制御ゲート実行、dynamic control flow対応SDKでの測定済み`Vector[Bit]`要素conditionのaddress resolution、個々の`Vector`要素に対する`expval` observableを堅牢化しています。

```
pip install qamomile==0.12.5
```

## 新機能

### Pauli Correlation Encoding(PCE)

`PCEConverter`は、`ommx.v1.Instance`またはQamomileの`BinaryModel`として与えられた二次QUBO/Ising最適化問題に対してPauli Correlation Encodingを実装します。公開されているPCEのsurrogateはpairwise interactionを仮定するため、高次のHUBO項は拒否されます。PCEは各スピン変数を圧縮されたレジスタ上の異なる`k`-body Pauli correlatorへ写像し、より大きな`num_qubits`を渡さない限り、`C(n, k) * 3**k >= num_variables`を満たす最小の`n >= k`を選びます(変数が0個の入力では`k`を返します)。割り当てられたobservableは`get_encoded_pauli_list()`で取得でき、最終的なcorrelator期待値は`decode(...)`で復号できます。`BinaryModel`入力は`BinarySampleSet`を返し、OMMX入力はOMMX自身のAPIで評価された`ommx.v1.SampleSet`を返します([#369](https://github.com/Jij-Inc/Qamomile/pull/369))。

```python
from qamomile.optimization.binary_model import BinaryModel
from qamomile.optimization.pce import PCEConverter

ising = BinaryModel.from_ising(
    linear={0: 1.0, 1: -1.0, 2: 0.5},
    quad={(0, 1): 2.0},
    constant=0.0,
)
converter = PCEConverter(ising, correlator_order=2)
observables = converter.get_encoded_pauli_list()
sampleset = converter.decode([0.8, -0.4, 0.2])
print(converter.num_qubits, len(observables))
print(sampleset.samples, sampleset.energy)
```

```text
2 3
[{0: 1, 1: -1, 2: 1}] [0.5]
```

20頂点の問題を3量子ビットで解くMaxCutワークフロー全体は、[Pauli Correlation Encoding(PCE)](https://github.com/Jij-Inc/Qamomile/blob/v0.12.5/docs/ja/algorithm/pce_maxcut.ipynb)を参照してください。

### QURI Partsサンプリングの再現性

`QuriPartsExecutor(seed=...)`と`QuriPartsTranspiler().executor(seed=...)`は、seedが指定されたときにseed付きのQulacsベースのsampling pathを使うようになりました。同じseedで作ったExecutorを同じ回路に対してサンプリングすると、同一のショットカウントが得られます。seedを省略すると従来どおり非決定的な経路を使います。このseedはQamomileのデフォルトQulacsベースsampler pathにのみ適用され、カスタムsamplerは既存の`sampler=`引数で引き続き差し替えられます([#441](https://github.com/Jij-Inc/Qamomile/pull/441))。

```python
from qamomile.quri_parts import QuriPartsTranspiler

transpiler = QuriPartsTranspiler()
executor = transpiler.executor(seed=42)
```

## バグ修正

- **測定済み`Vector[Bit]`要素conditionのaddress resolutionをdynamic control flow対応SDKで修正しました**。`s = qmc.measure(register)`の後に書く`if s[i]:`、`if s[0] & s[1]:`、`while s[i]:`、`if s[j:j+1][0]:`のようなスライスview上の条件を、Qiskitではsegmentationやemitのエラーではなく、正しいclassical bitアドレスへloweringします。CUDA-Qではscalarな`if s[i]:` pathも対象です。一方、QURI Partsはstatic measurementのSDKであり、runtime `if` / `while` control flowはサポートしません([#443](https://github.com/Jij-Inc/Qamomile/pull/443))。
- **`Vector`要素に対する`qmc.expval`がobservableを正しい物理量子ビットに束縛するようになりました**。`qmc.expval(q[1], obs)`や`qmc.expval((q[1],), obs)`は、誤った量子ビットアドレスへfallbackするのではなく、親`Vector`、スライスview、composite gateの結果、制御ゲートの結果をたどって解決されます([#450](https://github.com/Jij-Inc/Qamomile/pull/450))。
- **制御ゲートの実行がlocal SDK実装間でより一貫しました**。QURI PartsとCUDA-Qでは、制御化したユーザ量子カーネル、parameterized composite、powered control、broadcast target、選択されたcontrol indexなど、より多くの`qmc.control(...)`パターンをend-to-endで扱います。CUDA-Qはさらに、QURI Partsのguarded fallbackの対象外であるnative gate、stdlib composite、Pauli evolutionの制御ケースも扱います。拡張されたcross-backendテストは、samplingとexpectation-valueの両方の経路を実行します([#442](https://github.com/Jij-Inc/Qamomile/pull/442)、[#444](https://github.com/Jij-Inc/Qamomile/pull/444)、[#447](https://github.com/Jij-Inc/Qamomile/pull/447))。
- **`Vector`スライスのrefreshとborrow checkが壊れにくくなりました**。同じスライスのrefresh、隣接するsymbolic sliceのborrow、スライス代入のrefresh経路のうち安全性を証明できるものはコンパイルされます。一方、サポートされないsymbolic overlapのケースは引き続き拒否され、リポジトリ内の制限事項として明示されています([#444](https://github.com/Jij-Inc/Qamomile/pull/444))。
