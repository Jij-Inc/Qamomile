# Qamomile Algorithm Tutorial Generator

This skill turns a pair of inputs — *a reference describing an algorithm* and *the Qamomile function/class that implements it* — into a ready-to-ship tutorial file under `docs/en/`.

Output target: a single `.py` file in **jupytext percent format** that lives in Qamomile's documentation tree and is built by `jupyter-book` into the published docs site. The tutorial is *not* a generic notebook dropped elsewhere — it is a first-class doc artifact that has to match the conventions of the files already under `docs/en/tutorial/`, `docs/en/algorithm/`, `docs/en/usage/`, and `docs/en/integration/`.

## What "good" looks like in Qamomile's docs

A tutorial generated by this skill should let a reader who has never seen the algorithm:

1. Understand in one paragraph what the algorithm does and which Qamomile API it exposes.
2. See the problem stated precisely (objective, constraints, notation).
3. Follow the algorithm's logic step by step at a level above code.
4. See the Qamomile workflow end-to-end: `@qkernel` definition → `Transpiler.transpile()` → `executable.sample()` / `executable.run()` → `.result()` → decode.
5. See a concrete problem instance solved, with the output interpreted and (where dev deps allow) visualized.
6. Walk away knowing which higher-level Qamomile helper supersedes the from-scratch version (e.g. `qaoa_state`, `QAOAConverter`, `PCEConverter`, …) and where to go next.

Match the voice and pacing of [docs/en/algorithm/qaoa_maxcut.py](../../../docs/en/algorithm/qaoa_maxcut.py) and [docs/en/algorithm/qaoa_graph_partition.py](../../../docs/en/algorithm/qaoa_graph_partition.py). Those two files are the canonical templates — open one before you start writing, especially the first time in a session.

## Natural phrasing for quantum computing

Tutorials in this repository are written for an audience that already knows quantum computing's standard vocabulary. AI-generated prose tends to drift into phrasings that read as unnatural — or outright wrong — to that audience. Before drafting, calibrate against the conventions below and re-read the surrounding cell whenever something feels off.

**Use the field's standard verbs and nouns:**

- **Apply** a gate / operator (not "run", "execute", "send through", "use" a gate).
- **Measure** in a basis (not "read out"); the *result* is an *outcome* or a *bitstring*, not a "reading".
- **Prepare** a state (not "create", "make", "produce" a state for the initial preparation step).
- **Act on** a qubit / register (e.g. "$H$ acts on qubit 0"), not "modify" / "change".
- **Evolve** under a Hamiltonian (not "let the qubit evolve through"). Time evolution is "evolution under $H$".
- **Entangle** is a verb only between subsystems. Don't say "the state becomes entangled" without naming what it is entangled with.
- **Decohere** is intransitive ("the qubit decoheres"), not "the qubit gets decohered".
- A **circuit depth** is the longest critical path; "circuit length" is non-standard.

**Use the field's standard nouns:**

- **Qubit**, not "quantum bit" (except possibly once at first introduction). **Register** for a group of qubits.
- **Ancilla** (plural **ancillae** or **ancillas**) for helper qubits, with a sentence stating what role it plays.
- **Ansatz** (plural **ansätze** in formal writing, **ansatzes** acceptable). It is *a* parameterised state family, not "the ansatz function".
- **Hamiltonian**, **observable**, **expectation value**, **Pauli string**, **stabiliser/stabilizer**, **fidelity**, **state vector** — use exactly as the literature does. Don't paraphrase ("quantum energy operator", "average quantum value").
- **Shot** for a single sampled execution; **counts** for the dict of bitstring → frequency.
- **Bra**/**ket** notation $|\psi\rangle$ / $\langle\psi|$ for pure states; **density matrix** $\rho$ for mixed states. Do not write "the quantum state $\psi$" without the ket.

**Phrasings to avoid (they signal a non-native writer of the field):**

- "Quantum advantage" / "quantum supremacy" — only use when the *paper you are summarising* makes that specific claim. Never as flavour text.
- "Quantum parallelism" — imprecise and contested; describe what the algorithm actually does (interference, amplitude amplification, …) instead.
- "The qubit collapses" as a general statement — measurement projects the state onto an eigenspace of the measured observable; say *projection* or *post-measurement state* when precision matters.
- "Quantum probability" / "quantum randomness" — redundant. The probability of a measurement outcome is just a probability.
- "Quantum acceleration" — the standard term is **speedup** (polynomial, exponential, …).
- "Quantum gate operation" — pleonasm; "gate" suffices.
- "Solve in superposition" — superposition is a state, not an action.
- "Quantum noise" used as a catch-all — name the channel (depolarising, dephasing, amplitude damping, readout error, …) when it matters.
- Treating "Hamiltonian" as a synonym for "energy" — a Hamiltonian *is the operator*; its expectation value is the energy.
- Calling a controlled gate "a CNOT" when it isn't — `CX` / `CNOT` is specifically controlled-X. Use **controlled-$U$** for the general construction.

**Math and notation conventions:**

- Reduced Planck constant $\hbar$ is typically set to 1; if you keep it, say so once.
- Pauli operators: $X$, $Y$, $Z$ (not $\sigma_x$ / $\sigma_y$ / $\sigma_z$ unless the source paper uses them and switching would obscure the citation).
- Rotation gates include the standard $1/2$ factor in Qamomile: $\mathrm{RZ}(\theta) = e^{-i\theta Z / 2}$. The section guide's gate-convention call-out spells this out; never silently drop the factor.
- Use $\ket{0}, \ket{1}, \ket{+}, \ket{-}$ in math, and `|0>`, `|1>` in code/print statements — don't mix.

If a sentence feels stilted or over-formal, read it aloud as if explaining at a whiteboard. The tutorials in `docs/en/algorithm/` are deliberately conversational at the math level and terse at the code level — match that register.

## Inputs you should expect

The user provides:

- **One or more reference URLs** — typically an arXiv paper or a blog post that describes the algorithm. Fetch them with `WebFetch` and read enough to understand motivation, notation, and pseudocode.
- **A Qamomile API to demonstrate** — either a file path like `qamomile/optimization/pce.py` or a symbol like `qamomile.optimization.QAOAConverter`. Read the real source with `Read`. Note the signature of every public method you will call, the return types, and the data it expects. If a higher-level convenience helper exists alongside a lower-level one (e.g. `qaoa_state` vs. building the ansatz by hand), the tutorial should usually lead with the low-level construction and *finish* by showing the one-call equivalent — this mirrors the `qaoa_maxcut.py` pattern.

If either is missing, ask once before proceeding. Do not fabricate a paper or guess at a function signature you cannot see.

## Workflow

### 1. Read everything before writing anything

- `WebFetch` every reference URL. For long papers read the abstract, method section, and any pseudocode — you can skim experiments unless they motivate a parameter choice.
- `Read` the target Qamomile source file. List the public methods, their signatures, and what each returns.
- `Read` [docs/en/algorithm/qaoa_maxcut.py](../../../docs/en/algorithm/qaoa_maxcut.py) or the nearest existing tutorial in the category you will write into (see step 2). You are matching that file's cell rhythm, not inventing a new one.

### 2. Pick the destination directory

Qamomile has four content categories under `docs/en/`. Pick whichever fits the algorithm before writing:

| Destination | Scope | Filename convention |
|---|---|---|
| `docs/en/tutorial/` | Sequential SDK fundamentals — `@qkernel` basics, parameterized kernels, resource estimation, execution models, classical flow, transpilation. Not algorithm tutorials. | `NN_snake_title.py` with the next sequential `NN`. |
| `docs/en/algorithm/` | Concrete quantum algorithm examples — variational (QAOA, VQE, PCE), Hamiltonian simulation, state preparation / encoding, error correction, sample-based methods (QSCI), and converter-based optimization workflows (`QAOAConverter`, `QRAOConverter`, `PCEConverter`, …). Hand-wired and converter-driven algorithms both live here. | `<algo>_<problem>.py`, e.g. `qaoa_maxcut.py`, `pce_maxcut.py`, `qaoa_graph_partition.py`. |
| `docs/en/usage/` | How-to guides for individual Qamomile modules — *teaching one class or module* rather than running an end-to-end algorithm. | `<module>.py`, e.g. `binary_model.py`. |
| `docs/en/integration/` | Examples that use Qamomile together with an *external* library or quantum platform, including optional backends (qBraid, CUDA-Q, QURI Parts). The integration *is* the lesson. | `<backend>_<topic>.py` or `<integration>.py`, e.g. `qbraid_executor.py`. |

Rule of thumb:

- If the centrepiece is a **concrete quantum algorithm** — a variational ansatz, a hand-wired `@qkernel`, or a high-level `*Converter` that drives one end-to-end — it goes in `algorithm/`. This is true whether the example builds from scratch or calls a one-line converter.
- If the centrepiece is **teaching one Qamomile module** (how to construct a `BinaryModel`, how to use the resource estimator API, …) rather than solving a problem with an algorithm, it goes in `usage/`.
- If the centrepiece is **using Qamomile with an external platform** — an optional backend (`qamomile[cudaq-cu13]`, `qamomile[quri_parts]`, `qamomile[qbraid]`) or another library where the integration itself is the lesson — it goes in `integration/`.
- `tutorial/` is reserved for the numbered SDK-fundamentals series. New algorithm tutorials do **not** belong here.

### 3. Know what you are allowed to import

The tutorial ships with Qamomile and is built by `jupyter-book`. Two import tiers are available:

**Always allowed (Qamomile runtime dependencies, per [pyproject.toml](../../../pyproject.toml)):**

- `qamomile` (every public submodule: `qamomile.circuit`, `qamomile.optimization`, `qamomile.observable`, `qamomile.qiskit`, …)
- `jijmodeling`
- `ommx` / `ommx.v1`
- `qiskit`, `qiskit_aer`
- `sympy`
- Python standard library
- `numpy` — pulled in transitively by `qiskit`, used freely in existing tutorials.

**Allowed in `tutorial/`, `algorithm/`, `usage/` (doc-build dev dependencies, per `[dependency-groups].dev`):**

- `matplotlib`, `networkx`, `scipy`
- These are in the dev environment used by `jupyter-book build .`, so existing tutorials (e.g. `qaoa_maxcut.py`) import them freely. Use them where a plot or a graph layout teaches something the numbers cannot.

**Only allowed in `integration/` (gated on an optional extra):**

- `qamomile.cudaq` + `cudaq` → requires `qamomile[cudaq-cu13]`
- `qamomile.quri_parts` + `quri_parts.*` → requires `qamomile[quri_parts]`
- `qamomile.qbraid` + `qbraid` → requires `qamomile[qbraid]`

When a tutorial in `integration/` uses an optional backend, the first prose cell must tell the reader which extra to install. For tutorials in the other three categories, **do not** import optional backends — use `qamomile.qiskit.QiskitTranspiler` as the default backend, which ships with the core install.

### 4. Respect Qamomile's tutorial conventions

These are non-negotiable — every existing tutorial obeys them, and CI builds the full set.

**Jupytext header (verbatim):**

```python
# ---
# jupyter:
#   jupytext:
#     text_representation:
#       extension: .py
#       format_name: percent
#       format_version: '1.3'
#       jupytext_version: 1.19.1
#   kernelspec:
#     display_name: Python 3
#     language: python
#     name: python3
# ---
```

**Tags frontmatter (mandatory).** Every tutorial declares its tags inside a YAML block at the very top of the first markdown cell, **before** the H1 title. MyST consumes this block to generate the auto-rendered tag landing pages under `docs/en/tags/` (registered in `myst.yml` as `pattern: "tags/*.md", hidden: true`). The exact shape is:

```python
# %% [markdown]
# ---
# tags: [<category>, <topic1>, <topic2>]
# ---
#
# # <Tutorial title>
#
# <Abstract paragraph and step list go here.>
```

Rules for picking tags:

- **First tag = the directory category**, always. One of: `tutorial`, `algorithm`, `usage`, `integration`. This is the tag the category landing page filters on.
- **Add 0–3 topic tags** from the existing vocabulary below. Don't invent a new tag without good reason — the auto-generated tag pages only group what already exists, and a brand-new tag becomes a one-entry page that adds navigational noise.
- The total should be 1–4 tags. A pure-fundamentals SDK tutorial may carry only `[tutorial]`; a multi-faceted algorithm example may carry up to four.

**Existing topic-tag vocabulary** (read off the live tree as of writing — re-scan `grep -h "^# tags:" docs/en/**/*.py` if you need to confirm before adding):

| Tag | Meaning | Used in |
|---|---|---|
| `optimization` | Combinatorial or continuous optimization workflow (QUBO/Ising, JijModeling/OMMX problems). | `algorithm/qaoa_maxcut`, `algorithm/qaoa_graph_partition`, `algorithm/pce_maxcut`, `usage/binary_model`, `integration/qbraid_executor`. |
| `variational` | Variational quantum algorithm — parameterised ansatz + classical optimiser loop (QAOA, VQE, PCE). | `algorithm/qaoa_maxcut`, `algorithm/qaoa_graph_partition`, `algorithm/pce_maxcut`, `algorithm/vqe_for_hydrogen`. |
| `simulation` | Hamiltonian / time-evolution simulation (Trotter–Suzuki, exponentiation of $H$). | `algorithm/hamiltonian_simulation`, `tutorial/07_hermitian_decomposition`. |
| `encoding` | State preparation, amplitude encoding, Pauli encodings, problem-to-Hamiltonian maps. | `algorithm/mottonen_amplitude_encoding`, `tutorial/07_hermitian_decomposition`. |
| `error-correction` | QEC codes, stabilisers, syndrome decoding, transversal gates. | `algorithm/quantum_error_correction`, `algorithm/steane_code`. |
| `chemistry` | Quantum chemistry — molecular Hamiltonians, fermionic problems, VQE for molecules. | `algorithm/vqe_for_hydrogen`. |
| `primitive` | Low-level building blocks not tied to one algorithm (Trotter, multiplexers, Hermitian decomposition). | `algorithm/hamiltonian_simulation`, `algorithm/mottonen_amplitude_encoding`, `tutorial/07_hermitian_decomposition`. |
| `sample-based` | Methods that classically post-process measurement samples (QSCI, sample-based diagonalisation). | `algorithm/qsci`. |
| `resource-estimation` | Symbolic / pre-execution resource counts (gate, qubit, depth). | `tutorial/03_resource_estimation`. |

Before delivering, check that **the same tag set is reachable from at least one existing tutorial** — that confirms the tag is in the vocabulary and the new file will share its landing page with kin. If you genuinely need to introduce a new tag, surface the proposed name and rationale to the user before committing — it changes the navigation surface.

**First two cells:**

1. A markdown cell that **opens with the `tags:` YAML block described above**, then the tutorial title (`# <Algorithm> for <Problem>` or similar), a short paragraph summarising what the reader will build, and a numbered bullet list of the steps that make up the tutorial.
2. A code cell containing only the install hint, exactly as other tutorials have it:

   ```python
   # Install the latest Qamomile through pip!
   # # !pip install qamomile
   ```

   (The `# #` prefix keeps the `pip install` line commented out in the `.py` source but rendered as an executable magic in the `.ipynb` after `jupytext --to ipynb`.)

**End-to-end Qamomile workflow must appear in order:**

```
problem (math / JijModeling / BinaryModel)
  → encode / formulate (QUBO, Ising, or custom Hamiltonian via qamomile.observable)
  → @qkernel ansatz (or a Qamomile converter that builds one for you)
  → transpiler.transpile(kernel, bindings=..., parameters=[...])
  → executable.sample(executor, shots=..., bindings={...}).result()
    and/or
    executable.run(executor, observable=..., bindings={...}).result()
  → decode (converter.decode / BinaryModel.decode_from_sampleresult / custom)
  → interpret (print counts, visualise partition, compare to brute force)
```

The reader should see `transpiler.transpile(...)` and `executable.sample(...)` / `executable.run(...)` called with their real names in code cells — never hidden behind a helper.

**Heavy-compute cells must honour `QAMOMILE_DOCS_TEST`:**

For any cell that runs a classical optimiser or samples many shots, gate the expensive settings so the CI build stays fast:

```python
import os
docs_test_mode = os.environ.get("QAMOMILE_DOCS_TEST") == "1"
sample_shots = 256 if docs_test_mode else 2048
maxiter = 20 if docs_test_mode else 500
```

This is the pattern used in [docs/en/algorithm/qaoa_maxcut.py](../../../docs/en/algorithm/qaoa_maxcut.py) and [docs/en/algorithm/qaoa_graph_partition.py](../../../docs/en/algorithm/qaoa_graph_partition.py). Copy that shape; do not invent another knob.

**Markdown conventions:**

- Every markdown line starts with `# ` (hash + space). An un-prefixed line breaks the cell.
- Use `$...$` for inline math and `$$...$$` for display math. Hamiltonians, objectives, and update rules should be rendered, not code-fenced as plain text.
- Heading hierarchy is **strict**:
  - H1 (`# <title>`) — the tutorial title, once, inside the Abstract cell.
  - H2 — exactly these five labels, in order, and nothing else: `## Background`, `## Algorithm`, `## Implementation`, `## Run example`, `## Conclusion`. The Abstract has no H2 heading — it is the body of the first markdown cell under the H1 title.
  - H3 — natural topic names (`### What is MaxCut?`, `### Create the Graph`, `### QAOA Ansatz`, `### Decode and Analyze Results`, …). **Exception:** every H3 directly under `## Implementation` MUST be prefixed with `Step N:` (numbered sequentially from `Step 1:`), e.g. `### Step 1: Build the BinaryModel and PCEConverter`, `### Step 2: Transpile to an Executable Circuit`, … This signals the reader is following a fixed pipeline; the `Step N:` prefix does not appear under any other H2.
  - H4 — finer subsections inside an H3 when needed (`#### Step 1: …`, `#### Feasibility Check`, …).

**Paper citation conventions (MyST):**

The docs are built with MyST (`myst.yml`), which auto-renders bare DOI URLs into rich citation cards. **When citing an academic paper in body text, paste the DOI URL as a bare link — do not wrap it in `[text](url)` markdown link syntax, do not hand-type "(Farhi et al. 2014)", and do not use BibTeX/`{cite}` roles.** Qamomile tutorials currently rely solely on MyST's bare-DOI auto-citation; do not introduce a `references.bib` workflow.

The accepted form is:

```markdown
# QAOA was originally proposed by Farhi, Goldstone, and Gutmann
# (https://doi.org/10.48550/arXiv.1411.4028) as a hybrid quantum-classical
# heuristic for combinatorial optimization.
```

Notes:

- Always use the canonical `https://doi.org/<doi>` form. For arXiv preprints without a journal DOI, use the arXiv DOI (`https://doi.org/10.48550/arXiv.<id>`). Do not paste raw `arxiv.org/abs/...` URLs — MyST renders DOI URLs as citations but treats arXiv URLs as plain links.
- One DOI per cited claim; place it in parentheses right after the author names or claim it supports, like the example above.
- Where to cite: primarily in `## Background` (when the problem itself comes from a paper), `## Algorithm` (for the original method paper, encoding schemes, or compression-rate results), and the "Next steps" bullets in `## Conclusion`. Do **not** sprinkle citations through `## Implementation` — that section is about the Qamomile API, not the literature.
- **Plain markdown links remain correct** for non-academic references — blog posts, GitHub repos, Qamomile docs, vendor documentation. The bare-URL rule applies only to DOI-backed papers.
- **Never invent a DOI** for a paper the user did not provide and you did not fetch. If you cannot confirm a DOI from the source the user supplied, omit the citation rather than guess.

### 5. Draft the tutorial

A Qamomile tutorial has exactly six sections. `references/section_guide.md` defines what each section must contain, what must *not* appear there, and the H3 breakdown for each. Read it in full before writing the first tutorial in a session.

The six sections, in order, with their fixed H2 titles:

1. **Abstract** — no H2 heading. The Abstract cell contains the H1 title, a 2–4-sentence paragraph, and a numbered step list. Immediately followed by the install stub code cell.
2. **`## Background`** — the background of the *problem the algorithm solves* (e.g. MaxCut definition + concrete graph instance + optional brute-force reference). Not the algorithm itself.
3. **`## Algorithm`** — the paper-style conceptual walkthrough of the algorithm (ansatz, encoding, cost function, math, pseudocode). No Qamomile API calls in this section.
4. **`## Implementation`** — the Qamomile pipeline, step by step in H3s: QUBO → Ising, circuit definition, transpile, optimize, sample, decode. **Every H3 here MUST start with `Step N:`** (numbered from `Step 1:`) — see the heading hierarchy rule above. The target function must appear by name in a code cell here.
5. **`## Run example`** — analyse the final sample set: feasibility check, best solution, distribution, visualisation, and the "built-in equivalent" reveal when applicable.
6. **`## Conclusion`** — one markdown cell: recap (matching the Abstract's step list), 1–2 limitations, 1–2 next-step links.

Proportion length to complexity. Keep it under ~500 lines unless the algorithm truly warrants more.

### 6. Register the new file in the docs tree

The docs tree is now mostly **glob-driven**. Each category in `docs/en/myst.yml` registers its children with a pattern (`pattern: <category>/*.ipynb`), so a newly-added `.py` file is picked up automatically once jupytext builds the matching `.ipynb`. Tag landing pages are similarly auto-generated under `docs/en/tags/` and registered via `pattern: "tags/*.md", hidden: true`.

That leaves one human edit:

1. **The category's `index.md`** — [docs/en/algorithm/index.md](../../../docs/en/algorithm/index.md), [docs/en/usage/index.md](../../../docs/en/usage/index.md), [docs/en/integration/index.md](../../../docs/en/integration/index.md), or [docs/en/tutorial/index.md](../../../docs/en/tutorial/index.md). Add a one-line bullet under `## All articles` (or `## All tutorials`) matching the format that is already there:

   ```markdown
   - [My Algorithm](my_algo) — one-sentence description.
   ```

   The link target is the basename without an extension — MyST resolves it to the built notebook.

**Do not edit:**

- `docs/en/myst.yml` — unless you are adding a *new top-level category* (rare). The existing categories are pattern-matched and the new file appears automatically. Manually adding the file as a `- file: <category>/<name>.ipynb` entry breaks the pattern semantics and produces a duplicate entry in the TOC.
- `docs/en/index.md` — the main landing page lists categories, not individual tutorials, in its `## Sections` block. Don't add a bullet there.
- `docs/en/tags/` — entirely auto-generated from the `tags:` frontmatter on each tutorial. Writing a tag landing page by hand is unnecessary and gets overwritten on rebuild.

If the `category/index.md` edit is skipped the build still passes, but readers landing on the category page have no link to the new tutorial — they would have to discover it via the tag pages or the TOC sidebar. Always update the category index.

### 7. Sanity-check before delivering

Run through this mentally (or programmatically) before announcing the tutorial:

- Does every code cell run if executed top-to-bottom in a fresh kernel?
- Are all imports covered by the import tiers in §3 for the destination category?
- Does the first markdown cell open with a `# ---\n# tags: [...]\n# ---` block, with the category tag first and only tags drawn from the existing vocabulary?
- Does the tutorial actually call the target function by name (e.g. `QAOAConverter`, `qaoa_state`, `PCEConverter`) in a visible code cell? Use `grep` on the generated file to double-check.
- Are markdown cells uniformly `# ` prefixed with no fully-blank lines inside?
- Is `QAMOMILE_DOCS_TEST` wired up for any cell that runs > a few seconds of optimisation?
- Did you update the category's `index.md`? (`myst.yml` and `docs/en/index.md` should *not* need touching — see step 6.)
- Re-read the prose for unnatural quantum-computing phrasings (see the "Natural phrasing" section above). "Apply", not "run"; "qubit", not "quantum bit"; no "quantum parallelism" / "quantum advantage" as filler.

Programmatic check — convert to notebook without executing, to validate the percent format:

```bash
uv run jupytext --to ipynb docs/en/<category>/<filename>.py
```

If `jupytext` rejects the file, the percent format is malformed. If it accepts but produces an oddly structured notebook, a cell separator is likely misplaced. (The repository has both `.py` and `.ipynb` side by side for every tutorial — the `.ipynb` is the committed built artefact.)

Optional but recommended — confirm Python parses:

```bash
uv run python -c "import ast; ast.parse(open('docs/en/<category>/<file>.py').read())"
```

### 8. Tell the user what you changed

Report:

- The new tutorial path (clickable via markdown link).
- The `myst.yml` / `index.md` edits you made.
- A reminder that the Japanese translation is produced separately via the `/translate` skill once the English version is finalised — do **not** auto-generate the JA version from this skill. The translator keeps the two trees synchronised and needs the English to land first.

## What to avoid

- **Don't paraphrase the paper at length.** Re-explain in your own words at the right level. Quote no more than ~15 verbatim words from any single source.
- **Don't reimplement the target function from scratch and hide the Qamomile API.** Qamomile's functions must be called by their real name in visible code cells. A step-by-step hand-wiring is fine — and often good pedagogy — but it must be followed by the "using the built-in" section (§5 step 10).
- **Don't import optional backends outside `integration/`.** A tutorial in `tutorial/`, `algorithm/`, or `usage/` must run after a bare `pip install qamomile` plus the dev deps.
- **Don't invent citations.** Only cite papers/links the user provided or that you fetched and read.
- **Don't invent tags.** Pick from the existing tag vocabulary in §4. A new tag becomes a fresh single-entry landing page until enough tutorials adopt it — surface the proposal to the user first.
- **Don't skip the category `index.md` edit.** The new file appears in the TOC automatically, but the category landing page won't link to it without the bullet.
- **Don't edit `docs/en/myst.yml` or `docs/en/index.md` for a routine new tutorial.** Both are now category-level (pattern-based or section-level) and adding per-file entries breaks the convention.
- **Don't drift into unnatural quantum-computing phrasings.** "Quantum parallelism", "quantum advantage" as filler, "run a gate", "quantum bit", "quantum probability" — all flagged. See the "Natural phrasing for quantum computing" section near the top of this skill.
- **Don't use `/mnt/user-data/outputs/` or other sandbox paths.** This skill runs inside the Qamomile working directory; tutorials live at `docs/en/<category>/<filename>.py`.
- **Don't auto-generate the Japanese version.** Translation is a separate `/translate` workflow; running it from within this skill corrupts the pipeline.

## Example: canonical Qamomile tutorials

Two files are worth opening once at the start of every session — they define the target voice, cell granularity, math/code balance, and the "build-from-scratch then show the built-in" pedagogical move:

- [docs/en/algorithm/qaoa_maxcut.py](../../../docs/en/algorithm/qaoa_maxcut.py) — a hand-wired variational tutorial: problem → QUBO → spin model → hand-wired `@qkernel` ansatz → transpile → classical optimise → decode → visualise → compare with `qaoa_state`.
- [docs/en/algorithm/qaoa_graph_partition.py](../../../docs/en/algorithm/qaoa_graph_partition.py) — a converter-based tutorial: JijModeling problem → `QAOAConverter` → transpile → sample → decode, with `QAMOMILE_DOCS_TEST` gating.

If what you are about to write doesn't feel like one of those two, re-read whichever matches the destination category and adjust.
