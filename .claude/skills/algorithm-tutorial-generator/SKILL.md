# Qamomile Algorithm Tutorial Generator

This skill turns a pair of inputs — *a reference describing an algorithm* and *the Qamomile function/class that implements it* — into a ready-to-ship tutorial file under `docs/en/`.

Output target: a single `.py` file in **jupytext percent format** that lives in Qamomile's documentation tree and is built by `jupyter-book` into the published docs site. The tutorial must match the conventions of the files already under `docs/en/tutorial/`, `docs/en/algorithm/`, `docs/en/usage/`, and `docs/en/integration/`.

## Bundled resources

| File | When to read it |
|---|---|
| [references/tutorial_conventions.md](references/tutorial_conventions.md) | Before drafting. Jupytext header, tags, headings, citations, imports, `QAMOMILE_DOCS_TEST`. |

Treat these as the source of truth; this SKILL.md is the workflow that points at them.

## What "good" looks like in Qamomile's docs

A tutorial generated by this skill should let a reader who has never seen the algorithm:

1. Understand in one paragraph what the algorithm does and which Qamomile API it exposes.
2. See the problem stated precisely (objective, constraints, notation).
3. Follow the algorithm's logic step by step at a level above code.
4. See the Qamomile workflow end-to-end: `@qkernel` definition → `Transpiler.transpile()` → `executable.sample()` / `executable.run()` → `.result()` → decode.
5. See a concrete problem instance solved, with the output interpreted and (where dev deps allow) visualized.
6. Walk away knowing which higher-level Qamomile helper supersedes the from-scratch version (e.g. `qaoa_state`, `QAOAConverter`, `PCEConverter`, …) and where to go next.

Match the voice and pacing of [docs/en/algorithm/qaoa_maxcut.py](../../../docs/en/algorithm/qaoa_maxcut.py) and [docs/en/algorithm/qaoa_graph_partition.py](../../../docs/en/algorithm/qaoa_graph_partition.py). Open one before you start writing, especially the first time in a session.

## Writing Style

Use plain and direct technical English.

Avoid:

- em dashes (`—`)
- literary or promotional language
- exaggerated transitions
- metaphorical expressions
- motivational wording
- conversational filler

Examples of discouraged phrases:

- "Let's dive into"
- "powerful approach"
- "it is worth noting that"
- "navigate the complexities"
- "unlock the potential"
- "seamlessly"
- "robust and elegant"

Prefer:

- short sentences
- direct statements
- imperative instructions
- concrete technical descriptions

## Inputs you should expect

The user provides:

- **One or more reference URLs** — typically an arXiv paper or a blog post that describes the algorithm. Fetch them with `WebFetch` and read enough to understand motivation, notation, and pseudocode.
- **A Qamomile API to demonstrate** — either a file path like `qamomile/optimization/pce.py` or a symbol like `qamomile.optimization.QAOAConverter`. Read the real source with `Read`. Note the signature of every public method you will call, the return types, and the data it expects. If a higher-level convenience helper exists alongside a lower-level one (e.g. `qaoa_state` vs. building the ansatz by hand), the tutorial should usually lead with the low-level construction and *finish* by showing the one-call equivalent — this mirrors the `qaoa_maxcut.py` pattern.

If either is missing, ask once before proceeding. Do not fabricate a paper or guess at a function signature you cannot see.

## Workflow

### 1. Read everything before writing anything

- `WebFetch` every reference URL. For long papers read the abstract, method section, and any pseudocode — you can skim experiments unless they motivate a parameter choice.
- `Read` the target Qamomile source file. List the public methods, their signatures, and what each returns.
- `Read` [docs/en/algorithm/qaoa_maxcut.py](../../../docs/en/algorithm/qaoa_maxcut.py) or the nearest existing tutorial in the category you will write into (see step 2). You are matching that file's cell rhythm, not inventing a new one.
- `Read` [references/tutorial_conventions.md](references/tutorial_conventions.md), [references/section_guide.md](references/section_guide.md), and [references/quantum_phrasing.md](references/quantum_phrasing.md) in full before drafting.

### 2. Pick the destination directory

Qamomile has four content categories under `docs/en/`. Pick whichever fits the algorithm before writing:

| Destination | Scope | Filename convention |
|---|---|---|
| `docs/en/tutorial/` | Sequential SDK fundamentals — `@qkernel` basics, parameterized kernels, resource estimation, execution models, classical flow, transpilation. Not algorithm tutorials. | `NN_snake_title.py` with the next sequential `NN`. |
| `docs/en/algorithm/` | Concrete quantum algorithm examples — variational (QAOA, VQE, PCE), Hamiltonian simulation, state preparation / encoding, error correction, sample-based methods (QSCI), and converter-based optimization workflows (`QAOAConverter`, `QRAOConverter`, `PCEConverter`, …). | `<algo>_<problem>.py`, e.g. `qaoa_maxcut.py`, `pce_maxcut.py`. |
| `docs/en/usage/` | How-to guides for individual Qamomile modules — *teaching one class or module* rather than running an end-to-end algorithm. | `<module>.py`, e.g. `binary_model.py`. |
| `docs/en/integration/` | Examples that use Qamomile together with an *external* library or quantum platform, including optional backends (qBraid, CUDA-Q, QURI Parts). The integration *is* the lesson. | `<backend>_<topic>.py` or `<integration>.py`, e.g. `qbraid_executor.py`. |

Rule of thumb:

- If the centrepiece is a **concrete quantum algorithm** (variational ansatz, hand-wired `@qkernel`, or a high-level `*Converter`) → `algorithm/`.
- If the centrepiece is **teaching one Qamomile module** (constructing a `BinaryModel`, using the resource-estimator API, …) → `usage/`.
- If the centrepiece is **using Qamomile with an external platform / optional backend** → `integration/`.
- `tutorial/` is reserved for the numbered SDK-fundamentals series. New algorithm tutorials do **not** belong here.

### 3. Respect the tutorial conventions

The non-negotiable file-level shape (jupytext header, tags + topic-tag vocabulary, first two cells, end-to-end Qamomile workflow ordering, `QAMOMILE_DOCS_TEST` gating, markdown / heading rules, paper citation rules, and the import allowlist per category) is documented in [references/tutorial_conventions.md](references/tutorial_conventions.md). Read it in full before drafting.

### 4. Draft the tutorial

A Qamomile tutorial has exactly six sections in a fixed order. [references/section_guide.md](references/section_guide.md) defines what each must and must not contain, and the H3 breakdown for each.

Quick reminder of the six sections (full rules in `section_guide.md`):

1. **Abstract** — no H2. H1 title + **prose-only** 2–5-sentence summary (no list of any kind). Followed by the install-stub-and-imports code cell.
2. **`## Background`** — the problem the algorithm solves (definition + concrete instance). **Not** the algorithm itself, and **not** the brute-force / classical baseline (that moves to `## Result`).
3. **`## Algorithm`** — pure theory only. No Qamomile API calls, no implementation snippets, no gate-convention call-outs in this section.
4. **`## Implementation`** — Qamomile pipeline, step by step in H3s, each prefixed `### Step N:`. The target function must appear by name in a code cell. Qamomile-specific notes (gate factor of 2, parameter packing, etc.) live here.
5. **`## Result`** — analyse the final sample set: feasibility check, best solution, distribution, visualisation, "built-in equivalent" reveal. **Classical baselines (brute force etc.) live here**, next to the comparison they enable.
6. **`## Summary`** — one short markdown cell, **prose paragraphs only** (no list, no code). Summarise the algorithm and the Qamomile features used by name (specific gates, ansatz layers, `Transpiler`/`Converter`/`Executor` classes). **No "Limitations", no "Next steps"** subsections.

Proportion length to complexity. Keep it under ~500 lines unless the algorithm truly warrants more.

### 5. Copy-edit the English prose

After the draft is complete, do a dedicated copy-editing pass over every markdown cell **before** registering the file. Treat this as a separate step.

**Hard constraint: the overall structure stays exactly as drafted.** This pass changes prose *within* existing markdown cells only — no new/removed sections, no reordered content, no new/merged H3s, no new/removed code cells.

What this pass optimises for:

- **Accessibility to a non-engineer reader.** The audience includes domain experts (optimisation, chemistry, physics) who are *not* full-time software engineers. Prefer plain phrasing, define unavoidable framework terms inline at first use, and avoid software-engineering jargon (see "What to avoid" below).
- **Correct quantum-computing terminology.** Re-read every quantum-physics noun and verb against [references/quantum_phrasing.md](references/quantum_phrasing.md). Fix AI-typical phrasings that slipped into the draft.
- **Plain and direct technical English.** Apply the rules from the `## Writing Style` section near the top of this skill: use short sentences, direct statements, imperative instructions, and concrete technical descriptions.

Concrete moves:

- Read each markdown cell aloud (or in your head as if at a whiteboard). Tighten stilted or padded sentences.
- Strip filler ("It is important to note that …", "We can see that …", "This essentially means that …").
- Confirm every Qamomile API name in prose matches the actual call in the adjacent code cell.
- Confirm math symbols in `## Algorithm` reappear with the same letters in `## Implementation`.

If something genuinely structural surfaces here (a step in the wrong order, a value computed in the wrong section), stop the copy-edit, re-run step 4 to fix the structure, then resume.

### 6. Register the new file in the docs tree

The docs tree is now mostly **glob-driven**. Each category in `docs/en/myst.yml` registers its children with a pattern (`pattern: <category>/*.ipynb`), so a newly-added `.py` file is picked up automatically once jupytext builds the matching `.ipynb`. Tag landing pages are auto-generated under `docs/en/tags/` and registered via `pattern: "tags/*.md", hidden: true`.

One human edit is required:

- **The category's `index.md`** — [docs/en/algorithm/index.md](../../../docs/en/algorithm/index.md), [docs/en/usage/index.md](../../../docs/en/usage/index.md), [docs/en/integration/index.md](../../../docs/en/integration/index.md), or [docs/en/tutorial/index.md](../../../docs/en/tutorial/index.md). Add a one-line bullet under `## All articles` (or `## All tutorials`) matching the format already there:

  ```markdown
  - [My Algorithm](my_algo) — one-sentence description.
  ```

  The link target is the basename without an extension — MyST resolves it to the built notebook.

**Do not edit:**

- `docs/en/myst.yml` — unless adding a *new top-level category* (rare). Categories are pattern-matched; per-file entries break the pattern semantics.
- `docs/en/index.md` — lists categories, not individual tutorials.
- `docs/en/tags/` — auto-generated.

If the `category/index.md` edit is skipped the build still passes, but readers landing on the category page have no link to the new tutorial. Always update the category index.

### 7. Sanity-check before delivering

Run through this mentally (or programmatically) before announcing the tutorial:

- Does every code cell run if executed top-to-bottom in a fresh kernel?
- Are all imports covered by the import tiers in [references/tutorial_conventions.md](references/tutorial_conventions.md) ("Import tiers by destination category")?
- Does the first markdown cell open with a `# ---\n# tags: [...]\n# ---` block, with the category tag first and only tags drawn from the existing vocabulary?
- Does the tutorial actually call the target function by name (e.g. `QAOAConverter`, `qaoa_state`, `PCEConverter`) in a visible code cell? Use `grep` on the generated file to double-check.
- Are markdown cells uniformly `# ` prefixed with no fully-blank lines inside?
- Is `QAMOMILE_DOCS_TEST` wired up for any cell that runs > a few seconds of optimisation?
- Did you update the category's `index.md`? (`myst.yml` and `docs/en/index.md` should *not* need touching — see step 6.)
- Re-read the prose against [references/quantum_phrasing.md](references/quantum_phrasing.md). "Apply", not "run"; "qubit", not "quantum bit"; no "quantum parallelism" / "quantum advantage" as filler.

Programmatic check — convert to notebook without executing, to validate the percent format:

```bash
uv run jupytext --to ipynb docs/en/<category>/<filename>.py
```

If `jupytext` rejects the file, the percent format is malformed. If it accepts but produces an oddly structured notebook, a cell separator is likely misplaced.

Optional but recommended — confirm Python parses:

```bash
uv run python -c "import ast; ast.parse(open('docs/en/<category>/<file>.py').read())"
```

### 8. Tell the user what you changed

Report:

- The new tutorial path (clickable via markdown link).
- The `myst.yml` / `index.md` edits you made.
- A reminder that the Japanese translation is produced separately via the `/translate` skill once the English version is finalised — do **not** auto-generate the JA version from this skill. The translator keeps the two trees synchronised and needs the English to land first.

## What to avoid

- **Don't paraphrase the paper at length.** Re-explain in your own words at the right level. Quote no more than ~15 verbatim words from any single source.
- **Don't reimplement the target function from scratch and hide the Qamomile API.** Qamomile's functions must be called by their real name in visible code cells. A step-by-step hand-wiring is fine — and often good pedagogy — but it must be followed by the "using the built-in" subsection inside `## Result` (see the Result section of `references/section_guide.md`).
- **Don't import optional backends outside `integration/`.** A tutorial in `tutorial/`, `algorithm/`, or `usage/` must run after a bare `pip install qamomile` plus the dev deps.
- **Don't invent citations.** Only cite papers/links the user provided or that you fetched and read.
- **Don't invent tags.** Pick from the existing vocabulary in [references/tutorial_conventions.md](references/tutorial_conventions.md) ("Topic-tag vocabulary"). A new tag becomes a fresh single-entry landing page until enough tutorials adopt it — surface the proposal to the user first.
- **Don't skip the category `index.md` edit.** The new file appears in the TOC automatically, but the category landing page won't link to it without the bullet.
- **Don't edit `docs/en/myst.yml` or `docs/en/index.md` for a routine new tutorial.** Both are now category-level (pattern-based or section-level) and adding per-file entries breaks the convention.
- **Don't drift into unnatural quantum-computing phrasings.** "Quantum parallelism", "quantum advantage" as filler, "run a gate", "quantum bit", "quantum probability" — all flagged. See [references/quantum_phrasing.md](references/quantum_phrasing.md).
- **Don't lean on software-engineering jargon or abbreviations.** The audience includes domain experts who are not full-time software engineers, so opaque framework terms (e.g. "intermediate representation" / "IR", "AST", "DAG", "JIT", "ABI", …) actively harm readability. Rules of thumb:
  - Replace SW-engineering jargon with a domain-grounded description whenever you can ("Qamomile's internal circuit representation" instead of "Qamomile's IR").
  - If a term genuinely cannot be avoided, spell it out the first time and gloss it inline in plain language; never assume the abbreviation alone is enough.
  - When the term is purely Qamomile-internal (e.g. naming a private pipeline stage), the better fix is usually to drop it entirely.
- **Don't write low-density / redundant prose.** Filler phrasings like "It is important to note that …", "We can see that …", "This essentially means that …", "Let us now consider …" inflate length without informing. Every sentence should add something the previous sentence did not.
- **Don't use `/mnt/user-data/outputs/` or other sandbox paths.** This skill runs inside the Qamomile working directory; tutorials live at `docs/en/<category>/<filename>.py`.
- **Don't auto-generate the Japanese version.** Translation is a separate `/translate` workflow; running it from within this skill corrupts the pipeline.

## Example: canonical Qamomile tutorials

Two files are worth opening once at the start of every session — they define the target voice, cell granularity, math/code balance, and the "build-from-scratch then show the built-in" pedagogical move:

- [docs/en/algorithm/qaoa_maxcut.py](../../../docs/en/algorithm/qaoa_maxcut.py) — a hand-wired variational tutorial: problem → QUBO → spin model → hand-wired `@qkernel` ansatz → transpile → classical optimise → decode → visualise → compare with `qaoa_state`.
- [docs/en/algorithm/qaoa_graph_partition.py](../../../docs/en/algorithm/qaoa_graph_partition.py) — a converter-based tutorial: JijModeling problem → `QAOAConverter` → transpile → sample → decode, with `QAMOMILE_DOCS_TEST` gating.

If what you are about to write doesn't feel like one of those two, re-read whichever matches the destination category and adjust.
